# 复习 PDF 安全与可信流水线 v2

本文件定义面向真实考生、可供他人使用的安全边界。目标不是“多跑几次 AI”，而是用技术门禁把不可信输入、AI 幻觉、供应链风险、误发布和不可回滚问题隔离开。

## 1. 基本立场

1. AI 是不可信建议者，不是发布授权者。
2. 官方 PDF 默认采用 `source-faithful` 模式：只排版原图和用户已确认字段，不生成答案、解法、公式、定义、口诀或教材结论。
3. AI 生成或改写的任何学习内容只能进入候选区，不能自动进入正式 PDF。
4. AI 自报的 confidence、概率或“我已核对”不作为放行证据。
5. 任一检查不通过时必须 fail closed：保留上一版正式 PDF，不发布部分结果，不静默降级。
6. 不承诺零错误。系统只承诺可追溯、可阻断、可复核、可回滚。

## 2. 对照“AI 缸号图片整理器”的可靠性原则

- 两次独立识别对应两次独立审计；不能由同一轮生成过程自证正确。
- 标准化和正则校验对应严格 schema、路径、哈希、格式和资源上限校验。
- 两次结果出现实质冲突时强制人工处理；第三次结果只能作参考，不能用多数票掩盖冲突。
- 不采信 AI confidence。
- 不覆盖原文件对应正式 PDF 版本化、保留上一版和可回滚。
- 撤销对应发布指针可恢复到上一份已验证 manifest。
- API Key 不进普通配置和导出文件；正式 source-faithful 流水线根本不需要 AI Key。

## 3. 威胁模型

必须防护：

- 路径穿越、绝对路径、符号链接逃逸；
- 图片缺失、缺少 SHA-256、哈希不一致、扩展名伪装、损坏图片；
- 超大图片、解压炸弹、异常尺寸导致内存耗尽；
- 重复 ID、重复来源、类型冲突、未确认内容混入；
- 图片或备注中的 prompt injection；
- AI 对错题中的错误过程进行“合理化”并当成正确答案；
- 输入在审计和发布之间发生变化；
- 两份 PDF 与 manifest 部分更新；
- GitHub Actions 使用可变标签、依赖未锁定、脚本从网络临时下载；
- 工作流长期持有 `contents: write` 并直接推送 main；
- Secret 出现在日志、配置导出、产物或错误信息中；
- 新版本有误但覆盖旧版本，无法快速回滚。

## 4. 信任级别

### L0：原始来源

`data/index.json` 和 `data/assets`。仅 `organizationStatus=confirmed` 且 `kind` 为 `mistake` 或 `memory` 的记录可进入候选集合。

### L1：确定性候选

只包含：原图、标题、科目、知识路径、原始备注、已记录错因。所有文本原样转义，不做语义改写。不读取或展示 AI summary 作为事实。

### L2：AI 候选

仅允许做分组建议、重复候选识别或版面建议。AI 输出必须为结构化数据，且不能创建、删除或改写来源。L2 永远不能自动升级为正式内容。

### L3：正式发布

必须同时满足：输入快照一致、确定性构建通过、独立审计 A 通过、独立审计 B 通过、候选未被修改、发布者明确批准。正式发布不得在终审阶段临时修改内容。

## 5. 五阶段状态机

每轮使用不可重复的 `runId`，所有阶段通过仓库中的状态文件交接，禁止只依赖聊天上下文。

1. `SNAPSHOT`：冻结 main 的 commit SHA、index SHA、每张图片实际 SHA、策略 SHA。只读。
2. `BUILD`：在隔离候选分支生成 source-faithful PDF 和 page map。无 main 写权限。
3. `AUDIT_A`：独立重算来源、数量、顺序、哈希、每页映射和 PDF 基本结构。
4. `AUDIT_B`：独立检查视觉可读性、空白页、裁切、错配、异常文本和供应链信息。
5. `RELEASE`：只有人工批准且所有摘要完全一致时，才以单个 Git commit 发布；否则保留上一版。

状态只能按 `SNAPSHOT -> BUILD -> AUDIT_A -> AUDIT_B -> APPROVED -> RELEASED` 前进。缺阶段、跨阶段、摘要不一致或 main 已变化时全部阻断。

## 6. AI 内容规则

正式 PDF 中禁止 AI：

- 判断题目答案是否正确；
- 补写正确解法；
- 转写后自行修正公式；
- 根据错误过程推断教材结论；
- 生成“下次检查点”并当作事实；
- 把相似题强行合并为同一道题。

允许的 AI 候选操作：

- 给出可能重复的 source ID 列表；
- 给出可能的专题分组；
- 提出版面裁切建议。

两次独立 AI 结果只有在标准化后完全一致时，才可标为“稳定候选”；任何实质冲突进入 quarantine，人工处理。第三次结果不能覆盖前两次冲突。

## 7. 输入安全

- `imageSha256` 必填且必须是 64 位小写十六进制；缺失即阻断。
- 路径必须是仓库内相对路径；拒绝 `..`、绝对路径和任何符号链接。
- 通过文件魔数和 Pillow 解码双重确认格式，不只看扩展名。
- 设置单文件字节上限、宽高上限和总像素上限；Pillow decompression-bomb warning 按错误处理。
- 文本必须 HTML/XML 转义；控制字符清理；长度有硬上限。
- source ID 必须唯一并符合白名单正则。
- 输入快照后 main 发生变化，本轮候选自动失效。

## 8. 发布与回滚

- 构建阶段不得拥有 main 写权限。
- 正式文件以版本目录保存，并由 `generated/current.json` 指向当前版本；固定文件名仅作为兼容副本。
- 两份 PDF、page map、审计报告和 manifest 必须在同一个 commit 中更新。
- 发布前再次核对候选 commit、source digest、bundle digest。
- 发布失败不得清理上一版。
- 回滚只移动 current 指针或回退单个发布 commit，不重新生成内容。

## 9. 供应链与权限

- GitHub Actions 全局默认 `contents: read`；仅独立 RELEASE job 在人工门禁后获得最小写权限。
- Actions 必须固定到完整 commit SHA，不能只写 `@v4`、`@v5`。
- Python 依赖必须使用锁文件和哈希安装，禁止在发布任务里 `pip install --upgrade pip`。
- 禁止执行来自数据目录、图片元数据或 AI 输出的命令。
- 正式流水线不加载 AI Secret。AI 候选任务使用独立 Secret，日志中不得打印请求体、Key 或完整第三方响应。
- 对外分发前生成 SBOM、依赖清单和构建来源说明。

## 10. 默认发布策略

在 v2 完成并经过真实数据验收前：

- 停止旧的自动直推 main 流程；
- 只生成候选，不自动发布；
- 现有正式 PDF 保持不变；
- 任何 AI 增强内容都必须人工审阅；
- 至少连续三轮候选通过全部门禁后，再评估是否允许 source-faithful 模式自动发布。
